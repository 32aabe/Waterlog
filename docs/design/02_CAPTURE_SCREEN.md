# Waterlog Screen Specification

# 02. Capture (Memory Seed)

---

## Purpose

Capture is the beginning of every relationship in Waterlog.

Although users see it as a simple recording screen, internally it should always be treated as a **Memory Seed**.

Its purpose is not collecting observations.

Its purpose is not completing records.

Its purpose is to quietly preserve a meaningful moment before it disappears.

Every Capture becomes the first step toward a deeper relationship with a place.

---

# Core Philosophy

Capture is not where observations are stored.

Capture is where relationships begin.

The user is not filling out a form.

The user is leaving a small memory behind.

The experience should feel effortless.

The user should spend more time observing than recording.

---

# Emotional Goal

Capture should feel like writing a small note before the memory fades.

The user should feel:

- calm
- present
- unhurried
- attentive

Never:

- productive
- evaluated
- interrupted
- overwhelmed

The interface should disappear behind the act of remembering.

---

# Primary Question

Capture exists to answer only one question:

> **"What happened here today?"**

The place always comes first.

The user is remembering something that happened at a place,
not simply entering data.

---

# Information Hierarchy

## Primary

Memory

## Secondary

---

# Place First

Whenever possible,
the current place should already be recognized.

Instead of asking where the user is,
Waterlog quietly understands the location.

The interface begins with the place.

Example:

Library Pond

What happened here today?

---

# Memory

Writing should feel optional.

A memory may be:

- one sentence
- one word
- only a photo
- only presence

Silence is also a valid memory.

Waterlog should never pressure users into writing more.

---

# Photo

Photos are not evidence.

Photos are memory anchors.

Their purpose is not documenting birds.

Their purpose is helping users remember how a place changed through time.

Examples include:

- changing water levels
- seasonal vegetation
- reflections
- weather
- birds using water
- quiet moments

---

# Water Behavior

Water behavior is an important observation,
but it should never feel mandatory.

It acts as a gentle invitation toward closer observation,
not a required scientific field.

Users should gradually become better observers,
not better data collectors.

---

# Relationship Recognition

Waterlog quietly recognizes familiar places.

Instead of asking users to create a Spot,
the app notices repeated attention.

If a place begins to matter,
Waterlog may gently ask:

> "This place seems to matter to you."

> "Would you like to give it a name?"

Recognition should feel natural.

Never mechanical.

Never based on obvious counters.

---

# Completion

Capture should never end with:

✓ Saved

Instead,
the interaction should quietly acknowledge
that this moment has become part of the place.

Examples:

> This moment is now part of this place.

or

> This place now holds this moment.

The message should fade naturally after a brief pause.

---

# Interaction

The primary action should be represented by a simple icon,
not a large labeled button.

The action should feel like gently placing a memory down.

Avoid language such as:

- Save
- Submit
- Done
- Record

The interaction should feel almost invisible.

---

# Motion

Motion should be quiet.

The memory gently settles into the place.

Use:

- soft fade
- gentle downward movement
- subtle ripple

Avoid:

- success animations
- checkmark celebrations
- bouncing buttons
- confetti
- strong haptics

The user should feel closure,
not accomplishment.

---

# AI Behavior

AI should never interrupt Capture.

AI should never suggest what to write.

AI should never generate observations.

Its role is only to help organize memories later.

Capture belongs entirely to the user's own observation.

---

# Things to Avoid

Never include:

- Required fields
- Progress indicators
- Completion percentages
- Recording streaks
- Achievement systems
- Bird rarity prompts
- Long forms
- Scientific checklists

Capture should always remain lighter than observation itself.

---

# Success Criteria

A successful Capture makes the user think:

> "I'm glad I won't forget this."

rather than

> "I finished recording."

Later,
when returning to the place,
the user should feel that today's small memory became the beginning of something larger.

---

# Design Principle

Capture is not the end of observation.

It is the beginning of a relationship.

The purpose of Capture is not to create data.

It is to leave a small memory that allows a place to become meaningful over time.

Waterlog does not celebrate recording.

Waterlog celebrates remembering.
